## [Unreleased]

### Fixed
- Logs de sécurité non écrits en prod (ML-93) : `SecurityAuditLogListener` échouait sur toute exception serveur (≥ 500) avec `Permission denied` sur `/var/www/html/var/log`. Cause : le handler Monolog `security_audit` était le seul, en prod, à encore écrire sur un fichier (`var/log/security.log`) — `main`/`deprecation` écrivent déjà sur `php://stderr` précisément parce que `var/` (gitignoré) est absent de l'image et que le process PHP-FPM tourne en `www-data`, sans droit d'écriture sur `/var/www/html` (image buildée en root, pas de `chown`). `security_audit` aligné sur ce même pattern déjà en place plutôt qu'ajouter un volume/des droits Docker pour un fichier qui serait de toute façon perdu à chaque redéploiement d'un conteneur stateless

## [1.3.2] - 2026-07-22

### Added
- Couverture de tests backend (ML-123) : ajout de tests unitaires PHPUnit ciblant les points faibles identifiés dans le ticket, jusqu'au seuil de 70% sur les trois métriques (classes/méthodes/lignes). Sécurité d'abord — `App\Entity\FailedLoginAttempt` et `App\Security\AppUserChecker` passent à 100% de couverture méthodes/lignes (le cas nominal `checkPreAuth` était déjà couvert, il manquait le rejet d'un compte désactivé et le no-op `checkPostAuth`). Puis les entités identifiées dans le résumé du ticket comme faiblement couvertes (`Appointment`, `TreatmentSchedule`, `Treatment`, `TreatmentIntake`, `JournalEntryComment`, `Message`, `JournalEntry`), aucune n'ayant de test dédié auparavant — constructeurs, setters, et logique dérivée (`TreatmentIntake::markTaken`/`markNotTaken`, `JournalEntry::isEnteredByCaregiver`). Puis les branches restées non exercées dans les classes déjà partiellement testées (Voters `LiaisonInvitationVoter`/`MessageVoter`/`UserVoter` : jeton non authentifié ; `PasswordResetToken` : `getId`/`getCreatedAt` ; `AppointmentService`/`TreatmentService` : reprogrammation, remplacement des notes/horaires ; `MedicationReferenceService` : correspondance par "contains", cache de chargement, fichier JSON invalide ; `RegistrationService` : prénom/nom trop longs ; `AdminController` : corps JSON invalide ; `JournalExportController` : date au mauvais format ; `RegistrationController` : titre soignant renseigné). Enfin, pour passer le cap des 70% sur les trois métriques, ajout des DTOs (`AppointmentInput`, `AppointmentPatchInput`, `JournalEntryCommentInput`, `LiaisonInvitationInput`, `MessageInput`, `TreatmentInput`, `TreatmentPatchInput`), des exceptions métier (`DomainExceptionsTest`, data provider sur les 7 classes `Invalid*Exception`/`EmptyJournalExportException`) et de 8 State Processors API Platform jusque-là sans test (`AppointmentDeleteProcessor`, `MessageMarkReadProcessor`, `TreatmentIntakeToggleProcessor`, `LiaisonInvitationAccept/Reject/RevokeProcessor`, `MessageProcessor`, `JournalEntryCommentProcessor`) — construits avec de vrais Services (classes `final`, non mockables directement) et leurs seules dépendances (EntityManager/Repository/Security) mockées. Résultat : classes 35,64% → **70,30%**, méthodes 60,77% → **78,21%**, lignes 65,00% → **71,72%** (293 → 380 tests), toutes les classes exercées à 100% méthodes/lignes. L'écart restant vers 100% (Repository Doctrine, State Processors restants, Kernel) nécessiterait une vraie base de données ou sort du périmètre "tests unitaires, pas de BDD" du projet. Au passage, nettoyage des 15 dépréciations PHPUnit préexistantes (`->with()` appelé sur des stubs — n'a aucun effet et est déprécié depuis PHPUnit 12) dans `AdminControllerTest`, `AdminSupervisionControllerTest`, `AdminUserServiceTest` et `AccountServiceTest` : supprimé quand l'argument n'avait pas besoin d'être vérifié, remplacé par un mock local avec `expects()` quand le test avait réellement pour but de vérifier l'argument passé (ce qui n'était jusque-là pas le cas malgré le nom des tests, `with()` sur un stub étant silencieusement ignoré). Nécessitait un rebuild de l'image Docker `app` : l'extension PCOV ajoutée au Dockerfile dans un commit précédent du même ticket n'avait pas encore été construite
- Amélioration accessibilité RGAA/WCAG (ML-125), web et mobile : piège de focus (nouveau hook `useFocusTrap`, `frontend-web/src/hooks/useFocusTrap.js`) sur les boîtes de dialogue custom `role="alertdialog"` (avertissement d'expiration de session, confirmation de désactivation d'un utilisateur, confirmation de révocation d'une liaison) — focus posé sur le premier élément interactif à l'ouverture, Tab/Shift+Tab bouclé à l'intérieur au lieu de s'échapper vers le contenu sous-jacent, `aria-modal="true"` ajouté en cohérence. Zone de conversation (`MessagingPage`, web) passée en `aria-live="polite" aria-relevant="additions"` pour que l'arrivée d'un nouveau message soit annoncée par un lecteur d'écran sans relire tout l'historique ; côté mobile, `ConversationScreen` utilise `AccessibilityInfo.announceForAccessibility` pour annoncer les messages reçus via le polling pendant qu'une conversation est déjà ouverte (en excluant l'historique du premier chargement, via un suivi des identifiants déjà connus). `aria-label` explicite ajouté aux champs sans libellé visuel exploitable par un lecteur d'écran : recherche patient/médicament (`PatientAutocomplete`/`MedicationAutocomplete`, nouvelle prop `ariaLabel`), date/heure/type de consultation d'un RDV (`AgendaPage`), note libre du journal (`JournalPage`), dosage d'un traitement (`PatientJournalPage`)
- Icône MedLink dans l'en-tête, web et mobile (ML-124) : remplacement de l'emoji 🛡️ par le logo MedLink existant (`medlink-logo.png`, déjà utilisé sur l'écran d'accueil) dans `AppLayout` (web) et `Header` (mobile), pour une identité visuelle cohérente entre l'écran d'accueil et l'application connectée. Image traitée comme purement décorative (`alt=""` web, `accessibilityElementsHidden` mobile) puisque "MedLink" reste affiché en texte juste à côté

- Espacements insuffisants sur le formulaire de connexion web (ML-118) : l'écart entre le champ "Adresse e-mail" et le label "Mot de passe" en dessous, ainsi qu'entre le champ "Mot de passe" et le bouton "Se connecter", était de 20px — visuellement trop serré comparé à l'espacement label/champ (8px, jugé correct et pris comme référence). Cause réelle plus profonde qu'un simple ajustement de valeur : `.form-field` est une classe non scopée réutilisée à l'identique (même sélecteur, même spécificité) dans `LoginPage.css`, `RegisterPage.css`, `ResetPasswordPage.css` et `AccountPage.css`, toutes bundlées ensemble sans code-splitting par route — la cascade CSS ne retient que la dernière règle chargée dans le graphe de modules (`AccountPage.css`, importée en dernier dans `App.jsx`, `margin-bottom: 0`), qui écrasait silencieusement la valeur de `LoginPage.css` sur l'écran de connexion quel que soit son contenu. Un premier ajustement de la valeur dans `LoginPage.css` seule n'avait donc visuellement aucun effet (confirmé en inspectant `dist/assets/*.css` après build : une seule règle `.form-field{margin-bottom:0}` survivait à la minification). Corrigé en scopant les sélecteurs de l'écran de connexion sous `.login-card` (`.login-card .form-field`, etc.), ce qui augmente leur spécificité et garantit qu'ils gagnent la cascade indépendamment de l'ordre de chargement, sans toucher aux autres pages ni à leur collision préexistante (hors périmètre de ce ticket, limité à l'écran de connexion). Valeur retenue : 24px, cohérente avec les autres marges déjà utilisées sur cet écran (`login-subtitle`). Mobile non concerné (confirmé hors périmètre)
